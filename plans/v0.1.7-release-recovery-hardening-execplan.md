# EncourageLoop v0.1.7 Release Recovery Hardening ExecPlan

## Summary

After the v0.1.6 release, the project hit a realistic partial-release path: the release-preparation
commit and `v0.1.6` tag were pushed, `npm publish` was blocked by npm authentication and
publish-time OTP, the maintainer completed npm publication outside Codex, and Codex then verified
and completed the GitHub release and post-release evidence.

The goal is to make that recovery path deterministic and well documented for future releases. This
is release-process hardening only. Do not add a daemon, MCP server, app server, desktop overlay,
keyboard injection, LLM judge, network runtime behavior, automatic publish tool, automatic tag
repair, automatic release creation, or automatic retry logic.

## Phase R0 — Release Recovery Guidance

Deliverables:

- [ ] Add docs-contract coverage for release recovery guidance.
- [ ] Document how to distinguish npm login failure (`E401`) from publish-time OTP failure
  (`EOTP`).
- [ ] Document the partial-release state where a tag exists before npm or GitHub release
  completion.
- [ ] Document read-only verification commands before any retry.
- [ ] Document safe stop points and required maintainer confirmation for corrective actions.

Acceptance:

- [ ] Recovery guidance tells maintainers to verify npm versions, dist-tags, remote tags, and
  GitHub release state before retrying anything.
- [ ] Recovery guidance forbids automatic tag deletion, tag retargeting, force-push, unpublish, or
  publish retries without explicit maintainer approval.
- [ ] Recovery guidance keeps OTP handling manual and short-lived.
- [ ] Tests pin the guidance so it does not drift into automation.
- [ ] No release command is performed while preparing this guidance.
